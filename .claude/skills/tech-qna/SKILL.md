# /tech-qna

NEXTSTEP "나만의 Claude OS 만들기" 강의의 학습 보조 스킬. 강의 슬라이드(`*주차.pdf`)에 등장한 개념(스킬·에이전트·훅·오케스트레이터·MCP 등)이나 실습 중 마주친 기술 이슈를 받아서, 강의 자료에 충실하게 답하되 **혼자 검색해서 알아내기 어려운 부분**까지 메워준다.

이 스킬은 두 단계로 동작한다:
1. **답변 생성** — 메인 컨텍스트에서 직접 작성
2. **팩트 체크** — `fact-checker` 에이전트를 spawn해서 독립 컨텍스트로 검증, 결과를 답변 끝에 첨부

이 2단계는 `1주차.pdf` p13 "제가 경험한 AI (단점 위주) — 입장 번복·지침 무시·중도 포기"에 대한 방어 장치다. 같은 컨텍스트에서 자기 답을 다시 보면 자기 정당화에 빠지기 쉽다 — 그래서 p9의 "에이전트 = 독립 컨텍스트 보유"를 활용한다.

> 강의 자료(`1주차.pdf` p4) 학습 목표 3축을 답변 구조에 반영한다:
> **지식**(AI 엔지니어링 개념을 빠르게 따라잡기) · **경험**(직접 만들어보며 익히기) · **관점**(다양한 입장과 실제 사례 공유).

## 0) 질문 파싱

먼저 질문이 어느 카테고리인지 잡는다. 카테고리에 따라 참조할 소스가 달라진다.

| 카테고리 | 신호 키워드 | 주 참조 |
|---|---|---|
| 강의 개념 | 스킬, 에이전트, 훅, 오케스트레이터, MCP, SKILL.md | `*주차.pdf` |
| 실습 기술 | 명령어, 에러 메시지, 설정 파일 경로 | 프로젝트 코드 + 공식 문서 |
| 도구·생태계 | Claude Code vs Cursor, OMC, gh CLI, git 워크플로우 | 일반 지식 + 프로젝트 `CLAUDE.md` |
| 관점·선택 | "어느 쪽이 좋아?", "왜 이렇게?", trade-off | 다양한 입장 비교 |

애매하면 **사용자에게 한 번만** 되묻고, 두 번째부터는 추측해서 진행한다.

## 1) 강의 자료 우선 참조

질문이 강의 개념과 관련 있으면 해당 주차 PDF를 먼저 읽는다.

```bash
ls *주차.pdf
```

`Read` 도구로 PDF 페이지 범위를 지정해 인용한다. 10페이지 넘으면 `pages: "1-10"` 같이 분할해서 읽는다.

인용할 때는 **페이지 번호를 명시**한다 (예: `1주차.pdf p8 "스킬 / 에이전트 개념"`). 학습자가 슬라이드를 다시 펴서 확인할 수 있어야 한다.

## 2) 답변 구조

다음 5단 구조를 따른다. 짧은 질문이면 일부를 생략해도 되지만, **순서는 지킨다.**

### (1) 한 줄 결론

먼저 결론. "결론적으로 X입니다." 형태로 한 줄.

### (2) 무엇 · 왜 · 어떻게

CLAUDE.md 규칙(친절한 설명)에 따라:
- **무엇**: 정의·범위
- **왜**: 왜 이 개념·도구·선택이 등장했는지 (배경·동기)
- **어떻게**: 실제로 어떻게 쓰는지 (코드·명령·예시)

### (3) 다양한 관점

PDF 권장사항(p6 "같은 목표, 다른 접근·시각이야말로 큰 가치가 있음")을 반영해 **최소 2개의 입장**을 비교한다.

- A 입장: ... — 누가 / 어떤 맥락에서 / 어떤 이점
- B 입장: ... — 누가 / 어떤 맥락에서 / 어떤 이점
- 내가 추천하는 쪽 + 이유 (이건 명시적 의견으로 표시)

관점이 진짜 하나뿐이면 억지로 두 개 만들지 말고 "현재 업계 컨센서스는 X 한쪽으로 모입니다"라고 솔직히 적는다.

### (4) 실제 사례 / 실습 연결

- 이 프로젝트(`P98_Lecture_ClaudeCode-Os`)에서 직접 해볼 수 있는 명령·파일·경로
- 또는 사용자의 다른 강의 레포(`Lecture_Unity-game`, `Lecture_Docker-K8S`)에서 유사 패턴
- 또는 잘 알려진 공개 사례(이름과 출처를 함께)

### (5) 다음 단계

학습자가 바로 시도해볼 수 있는 액션 1~3개:
- 실행할 명령어
- 읽어볼 파일/문서
- 만들어볼 작은 스킬·에이전트 아이디어

## 3) 팩트 체크 — `fact-checker` 에이전트 spawn

답변 본문 (1)~(5) 작성이 끝나는 **즉시**, 별도 단계로 `fact-checker` 에이전트를 spawn한다. 답변 텍스트와 원 질문을 함께 넘겨 검증을 위임한다.

### 왜 매번 띄우나

`1주차.pdf` p11 "AI 시대의 협업 구조"에 따르면 AI는 **부분 예측 가능 / 비결정적**이다. 그럴듯한 환각이 섞일 수 있고, p13의 "지침 무시 · 중도 포기 · 입장 번복" 위험이 늘 있다. 답변과 검증을 같은 컨텍스트에서 하면 자기 정당화에 빠진다. 그래서 **독립 컨텍스트**가 필요하고, 이게 PDF p9의 "에이전트 = 독립 컨텍스트 보유"를 실전에서 쓰는 첫 사례다.

### 호출 방법

`Task` 도구로 호출:

```
Task(
  subagent_type="fact-checker",
  description="Verify tech-qna answer",
  prompt="원 질문: <사용자 질문>\n\n답변:\n<위 (1)~(5) 본문 전체>\n\n검증 범위: 전체"
)
```

호출이 시작되면 메인 대화에 에이전트가 spawn 되는 게 로그에 노출된다 — 학습자가 **"에이전트가 띄워지는 걸 실제로 보는"** 1주차 PRACTICE의 의도(`1주차.pdf` p14 03번)와 일치한다.

### 결과 첨부

에이전트가 돌려준 보고를 **답변 본문을 수정하지 않고** 끝에 그대로 덧붙인다. 형태:

```
---
**팩트 체크 (fact-checker 에이전트)**
[에이전트가 돌려준 ✅/⚠️/❌ 리포트]
```

- ❌가 1개 이상이면 학습자가 놓치지 않게 답변 첫 줄에도 한 줄 경고를 단다: "(주의: 아래 답변에 사실 오류 N건이 확인됐어요 — 팩트 체크 결과 참고)".
- ⚠️만 있고 ❌가 없으면 평소처럼 진행, 첨부만.
- ✅만 있으면 답변 끝에 한 줄 "팩트 체크 통과" 정도로 마무리.

### 생략 가능한 경우

- 질문이 **의견·취향**만 묻는 경우 ("어느 게 좋아?" 단독) — 검증할 사실 주장이 거의 없다.
- 답변이 **3줄 이하**로 짧고 사실 주장이 0–1개 — 그 자리에서 직접 한 번 더 확인하는 게 더 빠르다.
이 두 경우만 생략 가능. 나머지는 매번 spawn 한다.

## 4) 출처와 추측의 구분

- **확실한 사실**: 출처 명시 (`1주차.pdf p8`, `https://...`, `CLAUDE.md`, 프로젝트 파일 경로 등)
- **합리적 추론**: "추측입니다만 ..." 으로 시작
- **모르면**: 솔직히 "이건 강의 자료에도 없고 제 지식 컷오프 이후의 변경일 수 있어요. 공식 문서 X를 확인하세요"라고 안내. **그럴듯하게 지어내지 않는다.**

다음은 절대 금지:
- 존재하지 않는 명령어/플래그/파일 경로를 사실처럼 제시
- PDF에 없는 내용을 PDF에 있는 것처럼 인용
- "아마도" 없이 추측을 단정형으로 진술

## 5) 톤

- 강의 수강생(처음 Claude OS를 만들어보는 사람)이 청자
- 영어 약어는 **첫 등장 시 풀어쓰기** (예: "MCP(Model Context Protocol)")
- 비유는 환영. PDF p8이 "스킬 ≈ 클래스"로 비유한 것처럼, 학습자의 기존 멘탈 모델에 다리를 놓아준다.
- 길이는 질문 크기에 맞춘다. 한 줄 질문에 5단 풀세트로 답해서 학습자를 압도하지 않는다.

## 6) 출력 마무리

답변 끝에 한 줄로:
- "더 깊이 들어가고 싶으면 `<후속 명령>` 또는 `<후속 질문>` 해보세요"

이렇게 다음 대화를 열어둔다. 이 강의가 **적극적 소통**(`1주차.pdf` p6)을 권장하기 때문이다.

순서 정리:
1. 답변 본문 (1)~(5)
2. fact-checker 보고 첨부
3. 출력 마무리 한 줄

## 주의

- **코드 변경은 이 스킬의 책임이 아니다.** 질문에 답하다 보면 "고쳐줘"로 자연스럽게 넘어갈 수 있는데, 코드 수정 요청은 `/commit` 같은 다른 스킬·에이전트로 넘긴다. 이 스킬은 **설명·비교·제안 + 검증**까지.
- PDF에 등장하는 한국어 용어는 그대로 쓴다 (스킬, 에이전트, 오케스트레이터). 영문 강제 번역(Skill, Agent, Orchestrator)으로 학습자를 헷갈리게 하지 않는다 — 단, 검색 키워드 제공 차원에서 영문 병기는 OK.
- 사용자가 명백한 사실 오류를 지적하면 즉시 인정하고 정정한다. 우기지 않는다.
- **fact-checker의 보고를 임의로 손대지 않는다.** 답변에 유리한 쪽으로 ✅를 늘리거나 ❌를 빼면 안 된다. 에이전트가 돌려준 그대로 첨부. 강의 정신은 "잘하는 부분은 위임, 못하는 부분은 도와주기"(`1주차.pdf` p12)이지 "위임한 결과를 입맛에 맞게 편집"이 아니다.
- **fact-checker가 spawn에 실패하거나 비정상 종료**되면, 답변 끝에 한 줄로 "팩트 체크 미수행 (사유: ...)"이라고 정직하게 알린다. 조용히 건너뛰지 않는다.
